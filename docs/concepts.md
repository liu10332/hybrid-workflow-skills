# 核心概念

本文档介绍混合工作流（Hybrid Workflow）的核心概念和方法论。

## 什么是混合工作流？

混合工作流是一套结构化的软件开发方法论，结合了宏观管理和微观执行：

- **宏观管理**：里程碑→切片→任务三级结构
- **微观执行**：每个任务分解为2-5分钟的咀嚼式步骤
- **状态驱动**：基于文件的状态管理，支持中断恢复
- **质量保证**：强制TDD + 两阶段审查

## 核心理念

### 1. 结构化开发

将复杂的项目分解为可管理的部分：

```
项目
├── 里程碑（M001, M002, ...）
│   ├── 切片（S01, S02, ...）
│   │   ├── 任务（T01, T02, ...）
│   │   └── 步骤（Step 1, Step 2, ...）
│   └── ...
└── ...
```

**优势**：
- 清晰的进度跟踪
- 易于中断和恢复
- 便于并行开发
- 降低认知负荷

### 2. 咀嚼式执行

将每个任务分解为2-5分钟的小步骤：

```
任务T01：实现用户登录
├── Step 1：创建登录表单HTML（3分钟）
├── Step 2：添加CSS样式（2分钟）
├── Step 3：实现JavaScript验证（4分钟）
├── Step 4：编写测试用例（3分钟）
└── Step 5：运行测试（2分钟）
```

**优势**：
- 快速获得成就感
- 易于定位问题
- 减少上下文切换
- 提高代码质量

### 3. 状态驱动

基于文件的状态管理：

```
.hybrid/
├── STATE.md          # 当前状态（唯一权威源）
├── PROJECT.md        # 项目愿景
├── REQUIREMENTS.md   # 需求规格
└── milestones/
    └── M001/
        ├── M001-ROADMAP.md
        ├── M001-CONTEXT.md
        └── slices/
            └── S01/
                ├── S01-PLAN.md
                ├── S01-CONTEXT.md
                └── tasks/
                    └── T01-PLAN.md
```

**优势**：
- 支持中断恢复
- 便于协作
- 可追溯性
- 自动化友好

### 4. 质量保证

强制TDD和两阶段审查：

**TDD流程**：
1. 编写失败测试（RED）
2. 编写最小实现（GREEN）
3. 重构代码（REFACTOR）

**两阶段审查**：
1. 规格符合性审查
2. 代码质量审查

**优势**：
- 减少bug
- 提高代码质量
- 便于重构
- 自动化测试

## 工作流程

### 阶段1：需求分析（brainstorming）

**目标**：将想法转化为完整设计

**步骤**：
1. 探索项目上下文
2. 提出澄清问题
3. 提出2-3种方案
4. 分段呈现设计
5. 编写设计文档

**输出**：设计文档

### 阶段2：设计讨论（discuss-phase）

**目标**：收集用户偏好和决策

**步骤**：
1. 分析切片范围
2. 识别灰色区域
3. 逐一讨论决策点
4. 记录用户选择
5. 生成上下文文件

**输出**：切片上下文文件

### 阶段3：任务规划（writing-plans）

**目标**：创建详细实现计划

**步骤**：
1. 阅读切片上下文
2. 将切片分解为任务
3. 为每个任务创建计划
4. 分析任务依赖关系
5. 识别风险和缓解措施

**输出**：任务计划文件

### 阶段4：任务执行（subagent-driven-development）

**目标**：执行实现计划

**步骤**：
1. 阅读任务计划
2. 分解为咀嚼式步骤
3. 执行红绿重构循环
4. 进行两阶段审查
5. 更新任务状态

**输出**：实现代码、测试、任务总结

### 阶段5：切片验证

**目标**：验证切片完整性

**步骤**：
1. 运行切片级测试
2. 验证功能完整性
3. 生成用户验收测试
4. 更新切片状态

**输出**：切片总结、验收测试

### 阶段6：里程碑交付

**目标**：交付里程碑成果

**步骤**：
1. 验证里程碑成功标准
2. 生成发布说明
3. 打包交付物
4. 准备下一个里程碑

**输出**：里程碑总结、发布说明

## 状态管理

### STATE.md 文件

STATE.md是项目进度的唯一权威源：

```yaml
# 项目状态
project: "我的项目"
version: "1.0.0"

# 当前位置
milestone: M001
slice: S01
task: T02
phase: executing

# 进度跟踪
progress:
  milestone: 35%
  slice: 60%
  task: 80%

# 最后活动
last_activity: "2026-04-25T10:30:00Z"
last_action: "执行任务T02步骤3"

# 检查点
checkpoints:
  - timestamp: "2026-04-25T09:00:00Z"
    phase: planning
    description: "完成切片S01规划"
```

### 更新时机

必须更新STATE.md的时机：
1. 切片完成时
2. 里程碑切换时
3. 测试结果更新时
4. 发现新问题时
5. 会话结束前
6. 每完成3-5个任务后

### 恢复机制

中断恢复流程：
1. 读取STATE.md确定当前状态
2. 查找最近的检查点文件
3. 从检查点恢复执行
4. 更新状态文件

## 最佳实践

### 1. 保持STATE.md更新

STATE.md是恢复机制的基础，必须保持最新：
- 每次重要操作后更新
- 使用清晰的格式
- 包含所有必要信息

### 2. 分解任务为小步骤

每个任务应该分解为2-5分钟的小步骤：
- 易于执行
- 快速反馈
- 减少错误
- 便于测试

### 3. 强制TDD

每个任务都应该遵循TDD流程：
- 先写测试
- 再写实现
- 最后重构

### 4. 记录决策

所有重要决策都应该记录：
- 设计决策
- 技术选型
- 用户偏好
- 风险缓解

### 5. 定期检查点

定期创建检查点：
- 阶段完成时
- 重要决策后
- 风险缓解后
- 会话结束前

# M001: 项目骨架 上下文

## 切片描述
创建项目基础结构，包括目录、文档、Git仓库。

## 灰色区域分析

### 1. 目录结构
**决策点**：是否需要所有子目录立即创建？
- 选项A：创建所有目录（包括未来的示例项目目录）
- 选项B：只创建当前需要的目录
- 选项C：创建核心目录，其他按需创建

**用户选择**：待定

### 2. README.md内容
**决策点**：README应该多详细？
- 选项A：简洁版（项目介绍 + 快速开始）
- 选项B：详细版（包含所有技能说明）
- 选项C：分层（主README + 详细文档链接）

**用户选择**：待定

### 3. GitHub仓库设置
**决策点**：仓库应该公开还是私有？
- 选项A：公开（立即开源）
- 选项B：私有（开发完成后公开）
- 选项C：公开但标记为开发中

**用户选择**：待定

### 4. 许可证选择
**决策点**：使用哪种开源许可证？
- 选项A：MIT（最宽松）
- 选项B：Apache 2.0（包含专利授权）
- 选项C：GPL v3（强制开源衍生作品）

**用户选择**：待定

## 决策记录

### 锁定决策
- **D-01**: 使用单仓库结构
  - 选项：单仓库/多仓库/主仓库+子模块
  - 选择：单仓库
  - 理由：简单易维护，适合开源项目

- **D-02**: 保持现有Hermes技能格式
  - 选项：保持Hermes格式/转换为OpenClaw格式/两者都支持
  - 选择：保持Hermes格式
  - 理由：减少转换工作，保持兼容性

- **D-03**: 完整版包含所有技能
  - 选项：轻量级/完整版/模块化
  - 选择：完整版
  - 理由：提供完整解决方案

- **D-04**: 完整的开源项目模板
  - 选项：只有文档/文档+示例/文档+脚本/完整模板
  - 选择：完整模板
  - 理由：专业度高，用户体验好

### 延迟想法
- **I-01**: 添加CI/CD自动化
  - 原因：初期可手动测试
  - 后续：M005发布前添加

- **I-02**: 添加自动生成文档工具
  - 原因：初期可手动编写
  - 后续：根据社区反馈决定

### 系统自由裁量
- **C-01**: 目录命名规范
  - 约束：使用小写字母和连字符
  - 优先级：高

- **C-02**: 文件编码
  - 约束：使用UTF-8
  - 优先级：高

## 关键约束
1. 保持现有技能文件不变
2. 文档必须清晰易懂
3. 示例项目必须可运行
4. 安装过程必须简单

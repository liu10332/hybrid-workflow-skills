# Discuss Phase 切片讨论技能

在开始切片规划前，捕获用户对实现细节的偏好和决策。这确保规划阶段能准确反映用户愿景，而不是依赖默认假设。

**硬性门槛**：未完成切片讨论不得开始切片规划。讨论记录必须保存到切片上下文文件。

## 何时使用

每个切片开始规划前都必须使用此技能。即使用户说"你决定就好"，也要记录哪些决策留给了系统判断。

## 目标

1. **锁定用户偏好**：哪些决策必须按用户要求执行
2. **记录延迟想法**：哪些想法暂时搁置
3. **明确自由裁量**：哪些决策由系统判断

## 流程

### 1. 分析切片范围
阅读切片描述，识别灰色区域：
- **UI/UX功能**：布局、密度、交互、空状态
- **API/CLI功能**：响应格式、标志、错误处理、详细程度
- **内容系统**：结构、语气、深度、流程
- **组织任务**：分组标准、命名、重复、例外

### 2. 逐一讨论灰色区域
对于每个识别的灰色区域：
1. 解释为什么这是个决策点
2. 提出2-3种选项
3. 询问用户偏好
4. 记录决策或标记为"延迟"或"自由裁量"

### 3. 生成上下文文件
创建切片上下文文件，格式如下：

```markdown
# S{ID}: {切片名称} 上下文

## 切片描述
[来自切片计划的描述]

## 灰色区域分析
[识别的灰色区域列表]

## 决策记录

### 锁定决策
- **D-01**: [决策ID] [决策描述]
  - 选项：[考虑过的选项]
  - 选择：[用户选择]
  - 理由：[为什么这样选]
  
### 延迟想法
- **I-01**: [想法描述]
  - 原因：[为什么延迟]
  - 后续：[何时重新考虑]

### 系统自由裁量
- **C-01**: [决策描述]
  - 约束：[必须遵守的约束]
  - 优先级：[决策优先级]

## 关键约束
[必须遵守的技术或业务约束]

## 参考资料
[相关文档、标准、示例]
```

## 灰色区域识别指南

### UI/UX功能
- 布局偏好（网格 vs 列表 vs 卡片）
- 数据密度（详细 vs 简洁）
- 交互模式（点击 vs 悬停 vs 拖拽）
- 空状态处理（提示 vs 示例 vs 引导）
- 响应式设计断点

### API/CLI功能
- 响应格式（JSON vs XML vs 自定义）
- 错误处理方式（异常 vs 错误码 vs 混合）
- 分页策略（偏移 vs 游标 vs 全量）
- 认证机制（JWT vs OAuth vs API密钥）
- 版本控制策略

### 内容系统
- 内容结构（层次 vs 扁平 vs 网络）
- 写作风格（技术 vs 用户友好）
- 详细程度（完整 vs 摘要 vs 引导）
- 多语言支持策略

### 数据处理
- 数据验证规则
- 数据转换逻辑
- 错误恢复策略
- 性能优化点

## 示例对话

Agent：我准备为"传感器数据接入"切片创建实现计划。让我先了解一下你的偏好。

这个切片有几个需要决策的地方：

1. **数据验证规则**：传感器数据可能有缺失或异常值。
   - 选项A：严格验证，任何异常都拒绝
   - 选项B：宽松验证，标记异常但接受
   - 选项C：自适应验证，根据历史数据调整阈值
   你的偏好是？

2. **错误处理策略**：当传感器离线或数据无效时。
   - 选项A：立即告警，通知管理员
   - 选项B：记录日志，继续处理其他数据
   - 选项C：自动重试，超过阈值再告警
   你更倾向哪种？

3. **数据格式转换**：原始传感器数据需要转换为标准格式。
   - 选项A：实时转换，每次接收都转换
   - 选项B：批量转换，定时处理
   - 选项C：混合策略，关键数据实时，其他批量
   你希望用哪种方式？

用户：我选C、A、B。

Agent：好的，我记录下来：
- D-01：数据验证采用自适应验证
- D-02：错误处理立即告警
- D-03：数据转换采用批量策略

还有一个决策点：数据存储格式。你有偏好吗？还是让我根据性能考虑决定？

用户：你决定吧。

Agent：好的，我将根据性能考虑选择时序数据库格式。现在我来创建切片上下文文件。

[生成S01-CONTEXT.md文件]

## 与现有技能集成

### brainstorming技能
- discuss-phase是brainstorming的细化版本
- brainstorming处理项目级设计，discuss-phase处理切片级实现细节

### writing-plans技能
- discuss-phase的输出是writing-plans的输入
- 规划时必须遵守锁定决策，不能覆盖用户选择

### test-driven-development技能
- 讨论中确定的验收标准应转化为测试用例
- 测试应验证用户关心的决策点

## 煤矿防治水工程特定考虑

对于煤矿项目，额外关注：
1. **法规要求**：哪些决策受国家标准约束
2. **安全关键性**：哪些功能涉及安全预警
3. **集成约束**：与现有系统的兼容性要求
4. **性能要求**：实时性、准确性、可靠性指标
5. **维护性**：未来扩展和修改的便利性

## 注意事项
1. **控制时间**：讨论不要过于发散，聚焦关键决策
2. **记录完整**：所有决策都要记录，即使是"你决定"
3. **明确约束**：区分硬性约束和软性偏好
4. **保持灵活**：用户可能改变想法，记录修订历史
